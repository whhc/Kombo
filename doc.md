# 卡尔连招模拟器 (Invoker Combo Simulator) 开发方案 v1.1

> **版本说明:** 本文档由 v1.0 经设计共识 grill 修订而来。核心变更:产品形态由"后台全局键盘监听"改为"独立桌面模拟练习应用",移除 `rdev` 及相关的权限/VAC 议题;评估指标由三个独立公式改为"成功率 / 切球达成率 / 时长"三维模型;补充键位方案、预切起手、会话生命周期等此前未定义的设计。

---

## 0. 产品形态与定位

Kombo 是一个**独立的桌面按键练习应用**,供卡尔(Invoker)爱好者**日常快速练习切球/祈唤/释放的手感**。

- 玩家**不打开 Dota 2**,在 Kombo 窗口内按键,Kombo 自行模拟卡尔的元素球、祈唤与槽位逻辑,并给出练习反馈。
- 定位为"陪练工具",类比打字练习软件之于键盘 —— 重点是**重复、心流、复盘**,而非天梯考试。
- 产物为可安装的 Windows 桌面应用(打包 `.exe`),便于在卡尔爱好者社群中分发。

**形态选型结论(为何不采用全局键盘监听):** 早期方案设想用 `rdev` 全局钩子捕获玩家在 Dota 2 游戏内的按键,但该路径存在两个无法解决的硬伤:
1. 仅监听键盘**无法识别**玩家当局是否选了卡尔、是否在游戏中、是否在聊天框打字 —— 要识别必须读游戏内存,而读内存直接踩 VAC 红线。
2. 全局钩子引入 Windows UIPI 权限隔离与反作弊告知问题,复杂度高且与"轻量练习工具"定位不符。

故 v1 采用"窗口内 DOM keydown 监听 + 自模拟"形态,彻底绕开上述问题。

---

## 1. 技术栈与开发环境规范

* **宿主框架:** Tauri (v2) —— 用于打包可安装桌面应用、系统集成与本地存储。
* **前端渲染:** React + TypeScript + Tailwind CSS
* **键盘监听:** 前端 DOM `keydown` 事件(窗口内监听,**不使用 `rdev` 或任何全局钩子**)。
* **图表可视化:** ECharts / Recharts
* **本地存储:** IndexedDB(前端)或 SQLite(Rust 端)
* **推荐开发环境:** Windows 10,Node.js v25.9.0(建议通过 `fnm` 配置),Rust (stable)

> **关于"毫秒级无延迟":** 该表述在 v1.0 中针对全局监听场景。窗口内 DOM `keydown` 事件本身延迟在毫秒级,完全满足练习反馈需求,不再作为特殊要求单列。

## 2. 核心业务逻辑:卡尔技能机制

### 2.1 元素生成机制 (Orb Mechanics)

* **基础元素:** 冰 (Quas - Q),雷 (Wex - W),火 (Exort - E)。
* **状态维护:** 玩家头顶最多存在 **3 个**元素球。
* **进出队规则:** 先进先出 (FIFO)。当生成第 4 个元素时,最早生成的第 1 个元素被挤出。
* **冷却时间:** 无冷却。

### 2.2 祈唤机制 (Invoke - R)

* **合成判定:** 提取当前头顶的 3 个元素,**忽略排列顺序**(如 QWE 和 EWQ 视为相同)。
* **技能槽位:** 最多拥有 2 个已激活技能槽位,分别为"第一顺位"与"第二顺位"。
* **槽位更新逻辑:**
  * 如果新切出的技能不在当前槽位中:新技能进入第一顺位,原第一顺位技能被推至第二顺位,原第二顺位技能被移除。
  * 如果新切出的技能已在第一顺位:槽位顺序不变(游戏内仅重置冷却)。
  * 如果新切出的技能在第二顺位:该技能被提至第一顺位,原第一顺位技能降至第二顺位。

### 2.3 技能与键位字典表

| 技能名称 (中文/英文) | 元素组合 | 传统键位 | Dota 2 默认键位 (一顺位/二顺位) |
| --- | --- | --- | --- |
| 急速冷却 (Cold Snap) | Q - Q - Q | **Y** | **D** / **F** |
| 幽灵漫步 (Ghost Walk) | Q - Q - W | **V** | **D** / **F** |
| 寒冰之墙 (Ice Wall) | Q - Q - E | **G** | **D** / **F** |
| 电磁脉冲 (EMP) | W - W - W | **C** | **D** / **F** |
| 强袭飓风 (Tornado) | W - W - Q | **X** | **D** / **F** |
| 灵动迅捷 (Alacrity) | W - W - E | **Z** | **D** / **F** |
| 阳炎冲击 (Sun Strike) | E - E - E | **T** | **D** / **F** |
| 熔炉精灵 (Forge Spirit) | E - E - Q | **F** | **D** / **F** |
| 混沌陨石 (Chaos Meteor) | E - E - W | **D** | **D** / **F** |
| 超震声波 (Deafening Blast) | Q - W - E | **B** | **D** / **F** |

### 2.4 键位方案 (Keybind Scheme)

v1 支持两套**互斥**的键位方案,由图标主题选择决定当前激活方案:

```typescript
type KeybindScheme = 'LEGACY' | 'DOTA2';

// 方案 LEGACY(默认):技能 → 专属释放键(1:1 映射)
// 与 §2.3"传统键位"列一致:Tornado→X, ColdSnap→Y, ..., Meteor→D, ForgeSpirit→F, Blast→B
const LEGACY_KEYMAP: Record<SpellName, Key> = {
  ColdSnap: 'Y', GhostWalk: 'V', IceWall: 'G', EMP: 'C', Tornado: 'X',
  Alacrity: 'Z', SunStrike: 'T', ForgeSpirit: 'F', ChaosMeteor: 'D', DeafeningBlast: 'B',
};

// 方案 DOTA2:释放键 = D(第一槽位)/ F(第二槽位),由槽位状态机决定释放哪个技能
// 所有技能一视同仁,通过槽位语义释放(与 Dota 2 实际行为一致)
```

**键位方案与图标主题的绑定关系:**

| 图标主题 | 可选键位方案 | 默认 |
| --- | --- | --- |
| `DOTA1`(传统图标) | 仅 `LEGACY` | `LEGACY` |
| `DOTA2`(现代图标) | `LEGACY` 或 `DOTA2` | `DOTA2` |

> **关于 D/F 键冲突:** LEGACY 方案下 Meteor 的专属键是 D、ForgeSpirit 是 F,与 DOTA2 方案里 D/F 的槽位语义重名。由于两套方案**不会同时激活**(由图标主题选择决定),该冲突在 v1 不存在。自定义改键留待后续迭代。

### 2.5 释放键识别流程

玩家按下任意键时,按当前 `keybindScheme` 识别:

- **Q / W / E** → 切元素球(更新头顶队列)
- **R** → 祈唤(按 §2.2 合成判定,更新槽位)
- **释放键**(随方案不同):
  - `LEGACY`:按下键 → 查 `LEGACY_KEYMAP` 反查 → 释放对应技能
  - `DOTA2`:`D` → 释放第一槽位技能;`F` → 释放第二槽位技能(槽位空 = `MISS_CAST`)
- **其他键** → 忽略

> **v1 不模拟技能冷却 (cooldown):** 释放键总是释放当前槽位/对应技能,可快速重复释放,专注练习手部连贯性。冷却模拟与装备参数一同留待 v2。

## 3. 架构设计与数据流转 (Data Flow)

形态确定后,架构大幅简化 —— 无需独立的 Rust 监听线程与 IPC 推送,全部在前端完成:

1. **前端 DOM 监听:** Kombo 窗口内,React 根组件注册 `window.addEventListener('keydown', ...)`,捕获玩家在窗口内的按键。
2. **过滤层:** 过滤掉所有非相关按键,仅保留 `Q, W, E, R` 与当前方案下的释放键,附加 `performance.now()` 毫秒时间戳。
3. **React 状态机 (`useInvokerEngine`):** 收到合法按键事件,驱动内置状态机更新元素球队列、槽位、释放记录,产出 `ActionNode`。
4. **评估引擎 (Evaluator):** 会话结束后,将玩家实际释放序列与预设目标连招比对,按 §4 三维模型产出分析数据。
5. **持久化与可视化:** `ExecutionSession` 落盘 IndexedDB,Dashboard 从 IndexedDB 查询并喂给 ECharts 重绘。

```
玩家按键
   │  (窗口内 DOM keydown)
   ▼
过滤层 (仅留 Q/W/E/R + 释放键 + 时间戳)
   │
   ▼
useInvokerEngine 状态机 ──→ 元素球队列 / 槽位 / ActionNode[]
   │
   ▼ (会话结束后)
Evaluator ──→ 三维评估(成功率/切球达成率/时长)
   │
   ▼
IndexedDB 落盘 ──→ Dashboard / ECharts 复盘
```

> Tauri/Rust 层在 v1 仅承担:**打包可执行文件、系统集成(窗口/托盘)、可选的 SQLite 持久化**。键盘监听与状态机均在 React 前端完成。

## 4. 统计系统与算法模型 (Metrics)

### 4.1 核心数据结构

```typescript
// 基础元素 / 技能枚举
type Element = 'Q' | 'W' | 'E';
type SpellName =
  | 'ColdSnap' | 'GhostWalk' | 'IceWall' | 'EMP' | 'Tornado'
  | 'Alacrity' | 'SunStrike' | 'ForgeSpirit' | 'ChaosMeteor' | 'DeafeningBlast';

// 目标连招(用户在连招编辑器中创建)
interface TargetCombo {
  comboId: string;
  name: string;                                  // 如 "吹风磁暴陨石推波"
  spells: SpellName[];                           // 有序技能序列,允许重复
  preCastSlots: { d?: SpellName; f?: SpellName }; // 预切起手槽位,可空(空=从零开始练)
  // 约束:preCastSlots 中的技能必须是 spells 的前缀
}

// 单个操作动作记录
interface ActionNode {
  actionType: 'ORB' | 'INVOKE' | 'CAST' | 'MISS_CAST';
  key: string;             // 如 'Q', 'R', 'D'
  spellName?: SpellName;   // 实际合成/释放的技能名
  timestamp: number;       // 毫秒时间戳
  timeSinceLastMs: number; // 距上次操作间隔
}

// 单次连招练习会话
interface ExecutionSession {
  sessionId: string;
  comboId: string;
  status: 'SUCCESS' | 'FAILED';
  actions: ActionNode[];
  startTime: number;
  endTime: number;
  metrics: SessionMetrics;
}
```

### 4.2 三维评估模型

v1 采用**三个正交维度**评估一次会话,避免把"合成正确性"与"切球经济性"混在同一指标里。

#### 维度 ① —— 成功率 (Success Rate):二值,看合成对错与释放顺序

- **核心判定:** 技能**合成错误**(按 R 时头顶 3 球组合不是目标技能)或**释放顺序错误** → 整轮 `FAILED`。
- **成功节点:** 每个目标技能必须**按 spells 序列顺序,既合成(按 R 出现在槽位)又释放(按释放键放出)** 才算该步通过。只合成未释放、或释放了非当前目标 = 该步未通过 = 整轮 `FAILED`。
- **整轮 status:** 仅当所有目标技能都按序"合成+释放"完成,才为 `SUCCESS`;中途任一步出错 = `FAILED`。
- 单会话的 status 是二值;多轮统计的成功率(成功轮次/总轮次)留待 v2 配合挑战模式与成绩系统。

#### 维度 ② —— 切球达成率 (Optimal Orb Ratio):看切球经济性

- **只统计元素球切换 (Q/W/E) 次数**,不含 R、不含释放键 —— R 与释放是刚性必须的,数它们无意义。
- **最优切球数 `optimalOrbSwitches`:** 用**贪心近似**计算 —— 每切一个技能时尽可能复用当前头顶已有的球,从起手状态(含预切)推导到切完整条连招的最少切球次数。
  - 预切技能不计入分子分母(已在会话外完成)。
  - 精确最优化(枚举所有合法切法取最小值)留待 v2。
- **实际切球数 `actualOrbSwitches`:** 会话内玩家按 Q/W/E 的总次数。
- **公式:** `切球达成率 = optimalOrbSwitches / actualOrbSwitches × 100%`
- **超额切球数 `excessOrbSwitches = actualOrbSwitches − optimalOrbSwitches`:** UI 上标"多切了 N 次球",并可给出推荐切法。
- **FAILED 轮次:** 维度 ② 记为 N/A(目标未达成,切球数无意义)。

#### 维度 ③ —— 时长 (Duration):看速度

- **起点:** 玩家**第一个有效按键**的时刻。
- **终点:** **最后一个目标技能成功释放**的时刻。
- 单位毫秒。
- **FAILED 轮次:** 时长仍可记录(便于分析"卡在哪一步失败"),但不进入"成功轮次时长排行"。

#### 三维之间的关系

| 维度 | 看什么 | FAILED 轮次 |
| --- | --- | --- |
| ① 成功率 | 合成对错 + 释放顺序 | 决定本轮是否 FAILED |
| ② 切球达成率 | 切球经济性 | N/A(不算) |
| ③ 时长 | 速度 | 可记录,不进排行 |

三个维度**正交、不互相污染**:手快但常切错的玩家 ①低 ③好;切得准但绕远路的玩家 ①高 ②低 ③一般。

### 4.3 起手状态与预切的语义

`TargetCombo.preCastSlots` 表达"连招开始前已挂在 D/F 槽的技能"(实战中玩家预先切好等待):

- 可为空 = 从零开始练习(全部切球+祈唤计入本次手速)。
- 非空时,预切技能的"切球+R"**不计入**本次会话的切球达成率与时长统计,只算其释放键。
- **起手元素球不暴露给用户** —— 由评估器按"最小化后续切球"自动推导,编辑器无需提供 3 球选择器。
- 约束:`preCastSlots` 中的技能必须是 `spells` 的前缀(预切的就是连招前 N 个)。

## 5. UI 交互与数据可视化设计

### 5.1 界面模块划分

1. **连招编辑器 (Combo Builder):**
   - 提供技能拖拽面板,供玩家自定义练习连招(如:吹风 → 磁暴 → 陨石 → 推波)。
   - 允许设置**预切起手槽位**(`preCastSlots`,可选 D/F 各一个,从 `spells` 前缀中选)。
   - 允许切换图标主题(DOTA1/DOTA2)与键位方案(见 §2.4)。
   - 装备参数(A 杖/玲珑心影响前摇)留待 v2。

2. **主练习区 (Play Zone):**
   - UI 中间展示卡尔当前的 3 个元素球和 2 个技能槽位。
   - 底部显示目标连招的队列进度条,**温和提示**下一步目标技能(高亮当前步,不闪烁、不自动高亮该切哪几个球)。
   - **宽松继续模式:** 切错不中断会话,进度条标红错误步骤,玩家可继续按完整条连招,事后统一判定。

3. **数据复盘区 (Dashboard):**
   - 图表展示与历史过滤。

### 5.2 数据图表规范 (集成 ECharts)

* **按键节奏散点图 (Rhythm Scatter Plot):**
  * **X轴:** 相对时间线 (0ms 启动)。
  * **Y轴:** 两次按键之间的间隔延迟 (ms)。
  * **设计目的:** 直观暴露玩家在哪两个技能的切换之间出现了"大脑卡顿"或"手指粘连"。

* **成长趋势折线图 (Progression Line Chart) —— v2:**
  * 前端提供时间区段选择器(今日、近 7 天、近 30 天)。
  * **主Y轴:** 连招总耗时 (越低越好)。
  * **次Y轴:** 切球达成率百分比 (越高越好)。
  * **X轴:** 练习轮次/日期。

### 5.3 存储与数据过滤

所有 `ExecutionSession` 数据通过前端 IndexedDB 进行本地化存储。前端查询时,利用内置的时间选择器传递 `startDate` 和 `endDate` 参数,动态过滤数组并重新传递给 ECharts 实例进行重绘。
